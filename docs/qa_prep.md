# Q&A Preparation

This document prepares you for the hardest, most probing questions an examiner might ask about the technical decisions in this project. Study these to demonstrate depth of understanding beyond the slides.

---

### Q1: Why did you choose RAG instead of simply fine-tuning an LLM or a classification model on your specific ticket dataset?
**Answer:** Fine-tuning hardcodes knowledge into the model's weights. If our company introduces a new product or a new support queue tomorrow, a fine-tuned model must be retrained, which is computationally expensive and slow. RAG separates the knowledge base (the FAISS index) from the reasoning engine (the LLM). If we have new tickets or a new queue, we just embed them and add them to FAISS instantly. Furthermore, RAG drastically reduces hallucination because we force the LLM to generate answers based *only* on the retrieved context, not on its internal, generic training data.

### Q2: You used `all-MiniLM-L6-v2` for embeddings. Why that specific model? Why not something larger like OpenAI's `text-embedding-3` or a larger BERT variant?
**Answer:** It was a deliberate tradeoff between speed, compute footprint, and semantic quality. In our EDA, we found that the vast majority of our tickets are under 512 subword tokens, which perfectly fits the context window of `all-MiniLM-L6-v2`. It outputs a compact 384-dimensional vector, which keeps our FAISS index extremely lightweight and fast. Using a massive API-based model would introduce network latency for every single retrieval step, violating our requirement for fast, sub-millisecond triage, and a heavier local model would slow down inference without a proportionate gain in Retrieval@5 scores.

### Q3: How exactly does the system handle tickets it doesn't understand? What happens on a hallucination or a completely novel issue?
**Answer:** We implemented a `CONFIDENCE_THRESHOLD` (currently defaulting to 0.55). When a user submits a ticket, we compute the cosine similarity between the query and its nearest neighbors in FAISS. If the top result's score is below this threshold, it means the system has never seen anything like it, or the input is too short (e.g., "it's broken"). Instead of letting the LLM guess and hallucinate a confident-sounding but wrong answer, the system sets `needs_human_review = true` and skips automated generation entirely. We explicitly designed it to fail safely.

### Q4: Why use FAISS with `IndexFlatIP` instead of an approximate nearest neighbor (ANN) approach like HNSW?
**Answer:** Scale. FAISS `IndexFlatIP` performs exhaustive (exact) search via dot product. Because we L2-normalize our vectors first, dot product is mathematically equivalent to cosine similarity. We are searching across ~40,000 historical tickets with 384-dimensional vectors. This matrix fits entirely in memory and an exhaustive search takes fractions of a millisecond. Introducing ANN (like HNSW) would add complexity and index build-time for a microscopic gain in speed, while actually losing perfect retrieval recall. If our dataset scaled to 10 million tickets, we would switch to `IndexHNSWFlat`.

### Q5: You chose a Linear SVM for your baseline. What are its actual weaknesses compared to the RAG system?
**Answer:** The SVM relies on TF-IDF, which is a sparse, exact-keyword matching approach. If a historical ticket says "app crashed", and a new user submits "software unexpectedly closed", TF-IDF sees zero overlap because the words are entirely different. The SVM fails on synonyms. The RAG system uses dense semantic embeddings; `all-MiniLM-L6-v2` understands that "app crashed" and "software closed" occupy the exact same semantic space, allowing it to retrieve the right context even if the vocabulary doesn't match perfectly.

### Q6: Your presentation mentions optional cross-encoder reranking. Did you use it, and how does it measurably help?
**Answer:** We built the capability into the architecture but disabled it by default for speed. A bi-encoder (our main embedding model) is fast because it embeds the query and documents separately and compares them via a simple dot product. However, it misses deep relational nuances. A cross-encoder processes the query and the document *together* through attention layers, which is highly accurate but computationally heavy. When enabled, we use it only to re-score the top 10 results from FAISS. It measurably improves Retrieval@1 by resolving edge cases where two tickets use similar words but have different actual intents.

### Q7: If you were to deploy this to handle 10,000 tickets a minute in production, what would break first, and how would you fix it?
**Answer:** The external LLM generation step (Groq API) would be the first bottleneck, running into rate limits. The embedding and FAISS retrieval steps are local and highly scalable, but waiting on an external API is synchronous and slow. To fix it at scale, we would introduce an asynchronous queueing system (like Celery/RabbitMQ). The FastAPI backend would accept the ticket, immediately return a `202 Accepted`, process the retrieval and LLM generation in the background, and push the final result to the database or a webhook once complete.

### Q8: Be honest: your RAG F1 score (0.7223) is actually slightly *lower* than your baseline SVM (0.7311), AND they weren't even evaluated on the same sample size (n=250 vs n=4,217). Why should a business adopt the RAG system if the baseline is technically more accurate and evaluated more robustly?
**Answer:** That’s a fair critique, and we explicitly disclose that limitation. The baseline was evaluated on all 4,217 tickets because local SVM prediction is free and instant. Our RAG pipeline uses the external LLM (via Groq API) not just to write the answer, but to intelligently decide the queue based on the retrieved context. Running 4,217 tickets would hit harsh API rate limits, so we used a stratified n=250 sample. Even acknowledging the sampling variance, the 1.2% delta shows we've achieved functional parity. A business should adopt RAG because classification is only half the problem. The SVM only tells the agent the ticket belongs in "Billing." The human still has to read it, search the knowledge base, and type a reply. The RAG system sacrifices a negligible percentage of routing accuracy to instantly hand agents a pre-written, highly accurate draft response. That operational time savings vastly outweighs a 1% drop in initial routing accuracy.
